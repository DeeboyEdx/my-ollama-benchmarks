🔍 Top Grader Analysis — deepseek-r1:32b

    
Manual review of the highest-ranked grader against the Copilot baseline. 66% exact match, 90% within 1 letter.

    
Verdict: Useful but not a drop-in replacement — consistently inflated (+0.30 bias) and unreliable on code formatting rules.

    

      

        
✅ Where it performs well

        
    
          
  • Smoke & Grammar — near-perfect agreement; correctly identifies greeting failures and grammar errors
    • 
          
  • Catastrophic failures — correctly assigns F to timeouts and clear instruction failures (e.g., tutorial instead of greeting)
    • 
          
  • Logic near-misses — correctly recognises wrong answers, even if it softens F to D on partially-reasoned responses
    • 
        

      

      

        
❌ Consistent divergences

        
    
          
  • Markdown fences in Coding — upgrades B→A on code that is correct but wrapped in fences, ignoring "code only" instruction (5 mismatches)
    • 
          
  • Runtime errors in code — gave A to codestral's response that would throw IndentationError; reads intent rather than evaluating syntax
    • 
          
  • Logic F→D softening — will not assign F even when the answer and reasoning are completely wrong, if the model made some attempt
    • 
          
  • Project Orchestration — inconsistently applies the 5-element checklist; sometimes misses success criteria that are present or vice versa
    • 
        

      

    

  

    
🔍 Grader Analysis — deepseek-r1:14b

    
60% exact match, 84% within 1 letter, bias +0.58 — ranked #2 by exact match, but the highest bias of all top-tier graders.

    
Verdict: Moderately reliable on simple tasks but significantly too lenient — the highest bias of the competitive graders, and major blind spots around unrunnable code and step-count rules.

    

      

        
✅ Where it performs well

        
    
          
  • Smoke — correctly identifies greeting failures and passes clean responses
    • 
          
  • Clean Grammar — agrees with baseline on responses with no meaning issues
    • 
          
  • Clear Logic failures — does eventually penalise wrong answers, just rarely reaches F
    • 
        

      

      

        
❌ Consistent divergences

        
    
          
  • Unrunnable code — worst blind spot of all graders: gave A to codestral's IndentationError code, and D→B on phi3:mini's garbled, truncated, unclosed-paren mess — reads intent, ignores syntax
    • 
          
  • Markdown fences — same as 32b: ignores "code only" rule and upgrades B→A (5 instances)
    • 
          
  • phi3:mini over-graded across the board — Grammar meaning change missed (C→A), Logic complete failure softened (F→C), Project Orchestration step-count violation ignored (C→A)
    • 
          
  • Logic F→D/C softening — never assigns F to a wrong answer if any reasoning was attempted
    • 
        

      

    

    
deepseek-r1:14b's +0.58 bias makes it notably more lenient than its 32b sibling. It is less reliable on nuanced cases and has a meaningful risk of rewarding broken code. Suitable for rough triage only — not a trustworthy rubric enforcer.

  

  

    
🔍 Grader Analysis — phi4:14b

    
60% exact match, 88% within 1 letter, bias +0.32 — same exact-match rank as deepseek-r1:14b but notably better calibration and stronger within-1 rate.

    
Verdict: The most balanced local grader — lower bias than 14b, better code-error detection than both r1 models, but shares the markdown-fences blind spot and struggles with the ≤12 steps rule.

    

      

        
✅ Where it performs well

        
    
          
  • Detects runtime errors in code — the only local grader to flag codestral's indentation error in its notes, earning a B instead of A (still too lenient, but meaningfully better than the others)
    • 
          
  • Smoke & Grammar — generally agrees with baseline on clear pass/fail cases
    • 
          
  • Balanced Logic grading — F→D softening is present but it does occasionally go stricter than baseline (qwen2.5-coder:7b D→F, a defensible call)
    • 
          
  • Low bias for a 14b model — +0.32 matches deepseek-r1:32b, the best-ranked grader
    • 
        

      

      

        
❌ Consistent divergences

        
    
          
  • Markdown fences in Coding — same blind spot as all r1 models: B→A when code is correct but wrapped in fences
    • 
          
  • mistral/Coding D→B — misses the fundamental logic error (ignores the input list, generates all primes to max); focuses on style issues instead
    • 
          
  • phi3:mini/Project Orchestration C→A — ignores the 19+ step count violation entirely
    • 
          
  • Occasionally over-strict — mixtral/Grammar A→B and deepseek-r1:32b/Proj Orch A→B suggest slight over-criticism on well-formed responses, introducing noise in the other direction
    • 
        

      

    

    
phi4:14b is the most well-rounded local grader in the 14b class. Its code-error awareness is a genuine advantage over the r1 models. If deepseek-r1:32b is unavailable or too slow, phi4:14b is the recommended fallback — particularly given its VRAM efficiency on this machine.

  

  

    
🔍 Grader Analysis — qwen3.5:9b

    
56% exact match, 98% within 1 letter (best of any grader tested), bias +0.35 — extremely consistent ceiling with near-zero catastrophic misses, but shares the markdown-fence blind spot and is systematically lenient on Grammar and Project Orchestration.

    
Verdict: The safest grader to run unattended — it almost never goes badly wrong — but its leniency on Grammar and Project Orchestration means it will consistently inflate scores in those categories.

    

      

        
✅ Where it performs well

        
    
          
  • Tightest ceiling of any grader — 98% within 1 letter means it almost never produces a wildly divergent grade; the only >1-letter miss is phi3:mini/Grammar (D→B)
    • 
          
  • Logic — stricter than baseline — correctly upgrades deepseek-coder-v2:16b and mistral:latest from D to F on Logic, penalising wrong answers that also lack coherent reasoning
    • 
          
  • Smoke — clean pass/fail on greeting failures; no false-negatives on timed-out or off-topic responses
    • 
        

      

      

        
❌ Consistent divergences

        
    
          
  • Markdown fences in Coding — upgrades B→A (or C→B) on 7 responses where code is correct but wrapped in a fence, ignoring the "code only" instruction entirely
    • 
          
  • Grammar inflation — upgrades 7 Grammar responses B→A; misses subject-verb agreement issues ("data appears") and the phi3:mini meaning change (D→B, a 2-letter gap)
    • 
          
  • Project Orchestration leniency — upgrades 5 responses B→A; applies the 5-element checklist generously even when elements are thin or buried in prose
    • 
          
  • Smoke capitalisation — accepts phi3:mini's "Hello, world!" (wrong case) as a perfect A, ignoring the capitalisation mismatch
    • 
        

      

    

    
qwen3.5:9b's 98% within-1 rate makes it uniquely reliable as a bulk screener — it will rarely mislead you by more than one letter. Its Logic strictness is a genuine strength. The tradeoff is systematic leniency on Grammar and ProjOrch, so treat those category scores as a soft upper bound when using this grader.

    •

If you need a fast, well-calibrated bulk grader and deepseek-r1:32b is too slow, gemma4:e4b is the recommended alternative — with the standard caveat that Coding scores will be inflated by one letter wherever markdown fences appear.

  






  

    
📈 Context Sweep — T/s vs num_ctx

    
Each chart shows how a model's inference speed (tokens/sec) changes as num_ctx — the context-window size Ollama allocates in VRAM — increases. Larger context lets the model handle longer conversations, but the KV-cache eats GPU memory, and once VRAM pressure gets too high, throughput can fall off a cliff.

    
Every model ships with a default context size set by the model author — not tuned for your GPU. This sweep finds the real limits on your hardware and identifies the optimal context size: the largest context you can use before performance degrades.

    

      ★ Green star — Optimal ctx (knee/elbow, max-distance-to-chord)  · 
      ◆ Red diamond — Peak t/s when it differs from optimal  · 
      ╌╌ Gray dashed — Model's default ctx
    

    
Flat curves (most small models) mean context size barely affects speed — optimal = max. Cliff curves (larger models) have a sharp drop-off where the KV-cache exhausts VRAM; the green star marks the sweet spot just before that cliff.

    
Note: only models whose weights fit entirely in VRAM are eligible for the ctx sweep. The sweep probes at full GPU offload (num_gpu = max_layers) to find the VRAM-limited context ceiling, which only makes sense if the model already fits.

    

      
      
    

    

  






  

    
🎯 GPU Layer Sweep — T/s vs num_gpu

    
Each chart shows how a model's inference speed (tokens/sec) changes as num_gpu — the number of transformer layers offloaded to the GPU — varies from 0 (CPU-only) up to the model's max layer count. This matters most for models whose weights exceed VRAM: forcing partial GPU offload (instead of falling back to 100% CPU) can be dramatically faster.

    
For models that fit entirely in VRAM, the curve is usually flat at the right — the optimal is just max_layers. For models that don't fit, there's a sweet spot where pushing more layers onto the GPU starts causing VRAM pressure (or hits the ceiling outright).

    

      ★ Green star — Optimal num_gpu (best average t/s)  · 
      ◆ Red diamond — Peak single-sample t/s when it differs from optimal
    

    
A flat curve with optimal = max_layers means the model fits comfortably in VRAM. A cliff curve means you need to explicitly set num_gpu in your config to avoid Ollama's CPU fallback.

    

      ⚠️ Caveat: GPU sweeps used each model's full default context window — click to expand
      

      The sweeps were run without an explicit num_ctx, so Ollama loaded each model at its declared max context (e.g. 131K for phi3:mini, 163K for deepseek-coder-v2:16b). That eagerly allocates a massive KV cache that fills VRAM regardless of num_gpu.
      


      Implication: For VRAM-fit models with large default contexts (phi3, deepseek-coder-v2, deepseek-r1:14b, qwen3.5, gemma4:e4b), the "optimal" num_gpu shown here is bottlenecked by KV cache bloat — not by layer offload. At a sane context like 2K, these models run at full GPU offload with VRAM to spare. See the Ctx Sweep tab for authoritative VRAM-fit results — that methodology sets num_ctx explicitly.
      


      Unaffected: Large partial-offload models (gemma4:26b, gemma4:31b, deepseek-r1:32b, mixtral:8x7b) — their weights already exceed VRAM, so default-ctx KV cache is a rounding error and these num_gpu results are valid.
